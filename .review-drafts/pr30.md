## 🔴 必須 (CRITICAL): 編集ページに `requireManager()` がない

`app/(dashboard)/employees/[id]/edit/page.tsx` で `createClient()` を呼ぶ前に `requireManager()` を呼んでいません。RLS が DB アクセスを止めるので情報漏洩は起きませんが、未認証ユーザーは `/login` にリダイレクトされず `notFound()` の経路に流れます。プロジェクトの認証パターンが破れます。

```ts
// app/(dashboard)/employees/[id]/edit/page.tsx
import { requireManager } from '@/lib/auth/manager'

export default async function EditEmployeePage({ params }) {
  await requireManager() // ← 追加
  const { id } = await params
  const supabase = await createClient()
  // ...
}
```

`new/page.tsx` も同様に追加お願いします（フォーム送信は Action 側で守られていますが、レイヤごとに defense-in-depth でやる方針です）。

## 🔴 必須: archive / restore がDBエラーを握りつぶす

`archiveEmployee` / `restoreEmployee` は `update()` の戻り値の `error` を捨てているため、RLS 拒否やネットワークエラーが起きてもサイレントに成功表示されます。`createEmployee` / `updateEmployee` と同じパターンで error チェックしてください:

```ts
export async function archiveEmployee(id: string): Promise<void> {
  await requireManager()
  const supabase = await createClient()
  const { error } = await supabase.from('employees').update({ is_active: false }).eq('id', id)
  if (error) throw new Error('アーカイブに失敗しました')
  revalidatePath('/employees')
}
```

`useActionState` を使っていない簡易フォーム経由なので、`throw` で Next.js の error boundary に流すのが楽です。
