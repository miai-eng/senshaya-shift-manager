Thanks for the Stage 3 template settings page. `requireManager()` is called in both the page and the Action so security is solid, and errors are handled safely via `ERROR_MESSAGES` — nice work.

A few things I'd like fixed:

## 🔴 Required: cache is not refreshed after saving

`updateTemplate` in `actions.ts` doesn't call `revalidatePath` before `redirect()`, so stale content is shown after saving (Next.js's Full Route Cache persists).

```ts
// app/(dashboard)/settings/templates/actions.ts
import { revalidatePath } from 'next/cache'
import { redirect } from 'next/navigation'

// ... after the DB error check:
revalidatePath('/settings/templates')
redirect(`/settings/templates?saved=${type}`)
```

The login Action ([lib/auth/actions.ts](https://github.com/Tomoya300/senshaya-shiftmanager/blob/main/lib/auth/actions.ts)) follows the same pattern: `revalidatePath('/', 'layout')` before redirecting.

## 🟡 Recommended: silent failure when the seed hasn't been inserted

With `update().eq('type', type)`, if no matching row exists it returns `error: null` and reports success without writing anything. In a fresh environment where the migration's seed INSERT hasn't run, this goes unnoticed.

Switching to `upsert` makes it work regardless of whether the seed exists:

```ts
const { error } = await supabase
  .from('message_templates')
  .upsert({ type, body, updated_at: new Date().toISOString() }, { onConflict: 'type' })
```

## 🟢 Minor: narrow the type

Changing `type: string` to Supabase's generated type or the `'attend' | 'off'` literal union means a schema mismatch becomes a compile error.
