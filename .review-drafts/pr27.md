Stage 3 のテンプレート設定ページ、ありがとうございます。`requireManager()` がページ・Action 両方で呼ばれていてセキュリティはしっかりしているし、エラーメッセージも `ERROR_MESSAGES` で安全にハンドルされていて良いと思いました。

何点か修正お願いしたい点があります:

## 🔴 必須: 保存後にキャッシュがリフレッシュされない

`actions.ts` の `updateTemplate` で `redirect()` の前に `revalidatePath` を呼んでいないため、保存後に古い内容が表示されます (Next.js の Full Route Cache が残るため)。

```ts
// app/(dashboard)/settings/templates/actions.ts
import { revalidatePath } from 'next/cache'
import { redirect } from 'next/navigation'

// ... DBエラーチェックの後:
revalidatePath('/settings/templates')
redirect(`/settings/templates?saved=${type}`)
```

ログイン Action ([lib/auth/actions.ts](https://github.com/Tomoya300/senshaya-shiftmanager/blob/main/lib/auth/actions.ts)) も同じパターンで `revalidatePath('/', 'layout')` してから redirect しています。

## 🟡 推奨: シード未挿入時にサイレント失敗する

`update().eq('type', type)` だと、対象行がない場合 `error: null` で何も書き込まれずに成功扱いになります。新規環境でマイグレーションのシード INSERT が走らないケースで気付けません。

`upsert` に変えるとシード有無に依存せず動作します:

```ts
const { error } = await supabase
  .from('message_templates')
  .upsert({ type, body, updated_at: new Date().toISOString() }, { onConflict: 'type' })
```

## 🟢 軽微: 型を絞る

`type: string` を Supabase の自動生成型または `'attend' | 'off'` リテラルにすると、DBスキーマと食い違ったらコンパイルエラーで気付けます。
